## Recent main-branch work (this conversation)

### Customer row drag-and-drop (within and across tabs)
Two-state sort model on tabs: `manualOrder = false` (auto-sort) or `manualOrder = true` (user-arranged).

**Tab model additions:**
- `manualOrder: boolean` (default false)
- `sortColumn: string | null` (default null)
- `sortDirection: 'asc' | 'desc' | null` (default null)
- `setCustomerOrder(orderedCustomers)` — replaces array wholesale, length-checked
- All three new fields persist via toJSON/fromJSON; existing saves without them default cleanly

**Tab controller additions:**
- `dragInProgress` flag — `rebuildTable()` bails while set, prevents poll-mid-drag teardown
- `beginDrag()` / `endDrag()` — set/clear flag
- `#getDisplayOrder(tab)` — single source of truth for display ordering. Manual returns array as-is; auto sorts disconnected-first then ascending duration. All paths read from model, no textContent sniffing.
- `reorderCustomer(fromDisplayIdx, toDisplayIdx)` — within-tab; flips to manual; clears column state
- `moveCustomerToTab(radiusUsername, targetLabel)` — cross-tab; mode untouched on either side; appends to destination's customers array; toasts "already in target" on duplicate via NeonovaToast.error
- `toggleTabMode(label)` — flips manualOrder; on auto→manual freezes current display order as new array order; clears column state
- `#parseDurationToSeconds` is now dead (commented out); old textContent-sniffing sort retired

**Customer view additions:**
- `<tr draggable="true">`, `tr.dataset.username` set in constructor
- Drag handle (`⋮⋮`, U+2807) prepended to friendly-name cell as `<span class="drag-handle">`
- `dragstart` writes `application/x-neonova-customer` MIME with username, calls `tabCtrl.beginDrag()`, applies `neonova-row-dragging` class
- `dragend` cleans class and calls `endDrag()`
- Drag preempted if `#isEditing` (mid-rename)
- Inline `<style>` block injects `.drag-handle` styles (zinc-600 → emerald-400 on hover, cursor grab/grabbing)

**Tab view additions:**
- `dropIndicator` floating div (`position: fixed`, 3px emerald bar) appended to body, single instance
- `#injectStyles()` adds `.neonova-row-drop-indicator` and `tr.neonova-row-dragging` (opacity 0.4 + scale 1.02)
- `#handleDragOver` walks rows by `getBoundingClientRect()`, computes drop-above/below from midpoint, sets `lastDropTargetIdx`, positions indicator
- `#handleDrop` reads username from dataTransfer, finds source row index by walking tbody for matching `dataset.username`, calls `reorderCustomer`
- All handlers gated on `application/x-neonova-customer` MIME so other drags don't trigger
- Document-level `dragend` on tab view as cleanup catch-all for aborts

**Dashboard view (tab bar) additions:**
- Tab button layout: `[mode][bell][label][counts][×]`
- Mode glyph: `⇅` (auto) / `☰` (manual) — `☰` chosen over `⋮⋮` to avoid collision with row drag handle
- `.tab-mode-toggle` click → `toggleTabMode`, `stopPropagation()` so it doesn't switch tabs
- Each tab button gets BOTH a tab-reorder DnD listener set AND a customer-drop listener set, gated by `dataTransfer.types`. Tab reorder uses `text/plain`; customer drop uses `application/x-neonova-customer`. Source-order matters less than the type guards.
- `.customer-drop-target` class highlights destination tab teal (`#134e4a` background, emerald border) on customer-drag hover
- Active tab guards (`if (tab.isActive) return`) prevent treating active tab as a customer-drop target — tbody handles within-tab drops
- Document-level `dragend` cleans stuck `customer-drop-target` highlights
- `moveCustomerToTab` source-tab discovery uses `getActiveTab()` directly. Earlier "find first tab containing customer" version was buggy when the customer existed in multiple tabs — sourceTab could resolve to the destination, sourceTab===targetTab bailed silently before the duplicate check ran.

### Column-sort UI
Two-state model preserved — clicking a column header is just another way to enter manual mode. Column state is metadata about *most recent sort action*, not a live mode declaration.

**Behavior:**
- Sortable columns: Friendly Name, RADIUS Username, Status, Duration. Snapshot and Action: not sortable.
- Default glyph on every sortable column: `⇅` (same as tab auto-sort glyph — self-consistent vocabulary, "this can be sorted")
- Click sets that column's glyph to `↑` (ascending), all others revert to `⇅`. Click same column again: flips to `↓`. Click different column: previous reverts to `⇅`, new one shows `↑`.
- Clicking writes the sort through to the customers array and sets `manualOrder = true`. Tab glyph flips to `☰`.
- Reset paths (clear `sortColumn` and `sortDirection`):
  - Drag-reorder within tab
  - Tab mode toggle via tab icon (manual ↔ auto)
  - Cross-tab move and add-customer DON'T clear (cross-tab doesn't change source order; add-customer just appends to bottom and lets the next sort gather it)
- Each tab carries its own column state — switching tabs doesn't clear

**Sort comparators (`#getColumnComparator`):**
- friendlyName: `localeCompare` on `friendlyName || radiusUsername` (sort by what's displayed)
- radiusUser: `localeCompare` on `radiusUsername`
- status: status group primary (Connected/Connecting at top for asc, Disconnected/Error/Unknown at top for desc), `durationSec` ascending as tiebreaker (NOT flipped by direction — same as auto-sort tiebreaker)
- duration: numeric `durationSec`

**Implementation notes:**
- `buildTheadHTML(activeTab = null)` is parameterized — pure function of (columns, activeTab) → HTML. Defaults to ⇅ everywhere when no activeTab (initial render before `controller.load()` resolves). Static method.
- `renderTableHeader()` instance method finds `#content-area thead` and replaces it via `replaceWith()` on every `rebuildTable()` call. The `#content-area thead` selector is sufficient; the earlier comma-selector with `#header-container ~ #content-area` was overcomplicated.
- Click handling uses **event delegation** on `#content-area`, attached once via `#attachSortGlyphListener()`. Per-glyph `addEventListener` loop in `renderTableHeader` would have missed the inline initial render (which has glyph markup but no handlers). Delegation works regardless of how many times thead gets replaced. Pattern: `e.target.closest('.sort-glyph')`, read `dataset.column`, call `tabCtrl.sortByColumn(activeTab.label, columnKey)`.
- `data-column` on each glyph carries the column key.
- `SORTABLE` set inside `buildTheadHTML` is the single point of truth for which columns are clickable. Adding a column to the set is the only edit needed to make it sortable.

## Debugging episodes worth remembering
- **Twice during this conversation** my `web_fetch` of files from main returned a stale version that was missing real methods (`#evaluateAlerting`, `reorderTab`, `toggleNetworkTab`, etc.). The user pasted the correct current version both times. **Always ask the user to paste the current source before writing edits against a class with non-trivial existing logic.** Don't trust web_fetch as the canonical source.
- **Silent failures with no console errors** are usually delegation/handler attachment problems. Three common shapes: handler attached to wrong element, handler attached to element that gets replaced, handler never invoked because the call site doesn't exist. The error "X is not a function" surfaces only after the user gets to a code path that actually invokes X — so an edit that silently failed to apply may not produce an error until a different feature is exercised.
- **The userscript caching question is a real risk but always last on the diagnostic list.** The user's standing instruction is "do not blame the cache as first-line." Walk through actual code paths first.
- **Console queries returning 0 elements when the screen clearly shows them** can happen when devtools are attached to a different document context than the userscript runs in. Trust visual evidence over `querySelectorAll` in the page-context console for userscripts.

## Communication notes
- Hardware mode = design/planning, no code
- Software mode = code generation OK
- User holds strong MVC opinions, calls out architectural violations sharply
- "Dev is where the work gets done; main is the stable baseline."
- Branch dev → fix one discrete problem → merge to main → delete dev → re-branch. Don't bundle unrelated fixes.
- DO NOT pull `main.js` — production bootstrap is `scripts/neonova-dashboard.user.js`
- URL fetcher constraint: only accepts URLs the user has explicitly pasted in conversation OR that appeared in search results. If a URL fetch fails, ask user to paste the raw URL.
- When asked for "the methods" or edits to a class, prefer separate code blocks per method, not one big dump. When user asks for "the new class as a whole," do that, but err toward discrete edits for important classes the user doesn't want to risk corrupting on paste.
- When asked to edit a class, **first show edits cleanly without iterating over the same block multiple times**. Don't say "find this, replace with X, then find this again, replace with Y." Give each block in its final form once.
- When user is on dev/main/demo, REMEMBER which. Tailwind class availability differs:
  - Demo: precompiled tailwind.css with explicit safelist. Classes not in safelist produce no CSS.
  - Dev/main: runs as userscript on host page (admin.neonova.net). Tailwind from host or none.
- When making CSS changes that depend on Tailwind utilities, prefer owning rules in inline `<style>` blocks rather than relying on utility classes that may or may not exist.
- Long conversations degrade assistant accuracy — start fresh between phases.
- "Blame the cache" is a lazy diagnosis. Do not reach for it as first guess.
